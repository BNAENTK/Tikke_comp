# TTS 보이스 클로닝 + 저지연 읽기 연구 메모

Claude 관련 응답이나 채팅 메시지를 **보이스 클로닝된 음성으로**, `Edge TTS`나 `Google Speech`처럼 **빠르게 바로 읽기 시작하는** 구조를 만들기 위한 연구 메모.

## 목표

- 일반 TTS처럼 **응답 직후 바로 재생 시작**
- 가능하면 **보이스 클로닝** 지원
- 한국어 포함 다국어 품질 확보
- Hermes 또는 별도 TTS 파이프라인에 붙일 수 있어야 함

## 먼저 결론

### 1) 한 엔진으로 다 해결하려 하지 않는 게 맞다
가장 현실적인 구조는 **2단 분리**다.

- **초저지연 읽기용 엔진**: 속도 최우선
- **보이스 클로닝/고품질 엔진**: 음색 재현 최우선

즉,
- 실시간 채팅 읽기 = 저지연 TTS 엔진
- 특별 응답/긴 답변/하이라이트 = 보이스 클로닝 엔진

이 구조가 가장 실용적이다.

### 2) 완전 실시간 + 보이스 클로닝 + 한국어 + 싸고 쉬움까지 동시에 만족하는 단일 해답은 아직 드물다
현실적으로는 다음 셋 중 하나를 골라야 한다.

- **속도 최우선**
- **클론 품질 최우선**
- **로컬/비용 통제 최우선**

## 후보군 정리

### A. ElevenLabs
#### 장점
- 음질과 자연스러움이 매우 강함
- instant voice cloning / professional cloning 경로가 성숙함
- 다국어 지원 폭이 넓음

#### 단점
- 비용이 높은 편
- ultra-low-latency 전용으로만 보면 Cartesia류보다 불리할 수 있음
- 대량 실시간 읽기에는 비용 예측이 까다로움

#### 적합한 용도
- 중요한 응답
- 대표 보이스/브랜드 보이스
- 길고 퀄리티 중요한 낭독

### B. Cartesia
#### 장점
- 실시간/저지연 쪽에서 매우 강한 평가
- 실시간 voice agent 용도로 많이 언급됨
- time-to-first-audio가 매우 낮은 편으로 알려짐

#### 단점
- 언어/보이스 폭은 ElevenLabs보다 좁을 수 있음
- 클론 품질은 케이스별 검증 필요

#### 적합한 용도
- 즉시 반응해야 하는 채팅 읽기
- 음성 에이전트
- 짧은 응답의 빠른 재생

### C. PlayHT
#### 장점
- WebSocket 기반 실시간 TTS 시나리오에 자주 언급됨
- 고볼륨 처리와 실시간 스트리밍 쪽 장점

#### 단점
- 최상위 자연스러움은 ElevenLabs와 비교 검증 필요
- 실제 한국어 품질은 사전 테스트가 필요

#### 적합한 용도
- 실시간 스트리밍 TTS
- 비용/처리량 균형형

### D. MiniMax TTS
#### 장점
- Hermes 기본 지원 대상 중 하나
- 품질 평가가 좋은 편
- 다국어 대응력 기대 가능

#### 단점
- 보이스 클로닝/초저지연 실시간 읽기 요구를 동시에 만족하는지는 별도 검증 필요
- 즉시 스트리밍 UX는 엔진/API 형태에 따라 차이 큼

#### 적합한 용도
- Hermes에 비교적 자연스럽게 붙일 후보
- 일반 TTS 품질 상향

### E. 로컬/오픈소스 (Kokoro, XTTS, Chatterbox, RealtimeTTS 조합 등)
#### 장점
- 비용 통제 가능
- 로컬/프라이버시 강점
- 원하는 방식으로 파이프라인 커스터마이즈 가능

#### 단점
- 운영 복잡도 증가
- 한국어 품질, 클론 품질, 지연 튜닝을 직접 해야 함
- Windows/Hermes/gateway까지 붙이면 손이 많이 감

#### 적합한 용도
- 장기적으로 자체 TTS 스택을 갖고 싶을 때
- tikketone 같은 형제 프로젝트와 통합 실험할 때

## Hermes 관점에서 중요한 사실

Hermes 문서 기준으로 기본 TTS provider는 다음 계열이 있다.
- edge
- elevenlabs
- openai
- minimax
- mistral
- gemini
- xai
- neutts
- kittentts
- piper

즉 **기본 설정만으로는 “실시간 보이스 클로닝 전용 엔진”이 바로 들어 있는 구조는 아님**.
하지만 문서상 **custom provider / command provider 경로**가 있어서, 외부 TTS 서버나 Python 래퍼를 붙이는 확장이 가능하다.

## 내가 추천하는 현실적인 아키텍처

### 추천안 1 — 가장 현실적
#### “일반 읽기”와 “클론 읽기” 분리
- **일반 채팅 즉시 읽기**: Edge TTS 또는 Cartesia/PlayHT급 저지연 엔진
- **보이스 클론 읽기**: ElevenLabs 또는 전용 클론 엔진

#### 왜 좋냐
- 즉시성 유지 가능
- 보이스 클론 비용/지연을 모든 메시지에 강제하지 않음
- 짧은 메시지/긴 메시지에 서로 다른 정책 적용 가능

#### 추천 정책 예시
- 짧은 채팅, 시스템 메시지: Edge/저지연 엔진
- 중요한 답변, 하이라이트, 후원/이벤트: 클론 보이스 엔진

### 추천안 2 — 단일 상용 엔진 중심
- ElevenLabs 하나로 통일하거나
- Cartesia 하나로 통일

#### 장점
- 단순함

#### 단점
- 비용/지연/품질 어느 한쪽에서 아쉬움이 생길 확률 큼

### 추천안 3 — 로컬 TTS 서버 + Hermes 커스텀 provider
- RealtimeTTS 같은 스트리밍 래퍼
- 뒤에 XTTS/Chatterbox/기타 클론 모델 연결
- Hermes는 command provider나 별도 브리지로 호출

#### 장점
- 완전 커스텀 가능
- 장기적으로 가장 강력함

#### 단점
- 구축 난이도 최고
- 운영과 디버깅 비용 큼

## “Edge TTS처럼 바로 읽기 시작”하려면 필요한 조건

### 필수 조건 1. 스트리밍 또는 chunk 기반 재생
한 번에 mp3 파일이 완성된 뒤 재생하는 구조면 느리다.
필요한 건:
- WebSocket streaming
- chunked audio output
- time-to-first-audio 낮은 API

### 필수 조건 2. LLM 응답 완료를 기다리지 않고 문장 단위/청크 단위 공급
진짜 빠르게 들리게 하려면:
- 답변 전체 완성 후 TTS 시작 ❌
- 문장/절 단위로 먼저 도착한 텍스트부터 TTS 투입 ⭕

### 필수 조건 3. 재생 파이프라인 분리
- 텍스트 생성 스레드
- TTS 생성 스레드
- 오디오 재생 스레드
를 분리해야 체감 지연이 줄어든다.

## 보이스 클로닝 쪽에서 봐야 할 체크포인트

- instant clone 지원 여부 (짧은 샘플로 가능한지)
- 한국어 음질
- 스트리밍 재생 가능 여부
- API/WebSocket 지원 여부
- 비용 구조
- 상업 사용 / 동의 / 저작권 / 워터마크 정책

## 구현 방향 추천

### 1단계
- Hermes 기본 TTS는 유지
- 별도 외부 TTS 서버 후보 2개 비교
  - ElevenLabs
  - Cartesia 또는 PlayHT

### 2단계
- “일반 읽기용” 저지연 엔진 1개 선정
- “클론 전용” 엔진 1개 선정

### 3단계
- Hermes에서 커스텀 provider 또는 command provider로 연결
- 조건 분기:
  - 일반 메시지 → 저지연 엔진
  - 특정 이벤트/긴 답변 → 클론 엔진

## 내 추천 결론

### 가장 추천
- **일반 읽기**: Edge 유지 또는 Cartesia/PlayHT급 저지연 엔진
- **클론 음성**: ElevenLabs 우선 검토

이유:
- 사용자 체감 속도와 음색 품질을 동시에 잡기 가장 쉬움
- 모든 메시지를 클론 엔진으로 읽게 하는 것보다 실용적임

### Hermes 안에 바로 붙일 때의 현실적 순서
1. 먼저 **상용 API 2개**로 속도/한국어 품질 비교
2. 그다음 Hermes custom provider로 붙임
3. 마지막에 필요하면 로컬 TTS 서버로 확장

- 실제 검증 항목:
  - 한국어 자연스러움
  - 첫 음성 시작 시간

  - 클론 음색 유사도
