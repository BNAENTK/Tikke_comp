# Tikke Claude Code 프롬프트 템플릿 — TTS 보이스 클로닝/저지연 구현편

`tikke` / `tikketone` / Hermes 연동 관점에서 **보이스 클로닝 + 저지연 TTS**를 `Claude Code`에 맡길 때 쓰는 실전 프롬프트 템플릿.

## 핵심 원칙

- 처음부터 "다 만들기"를 시키지 말고 **조사 → 설계 → 스파이크 → 본 구현**으로 나눈다.
- `Claude Code`가 마음대로 벤더를 확정하지 못하게 **비교 기준**을 먼저 준다.
- "빠르게 말하기"를 추상적으로 말하지 말고 **TTFA, 스트리밍, chunk 재생, 한국어 품질, 보이스 클론 분기 조건**으로 구체화한다.
- 처음 프롬프트에서 **완성 구현**보다 **검증 가능한 산출물**을 요구해야 한다.
- 오버스코프를 막기 위해 `tikke` / `tikketone` / Hermes 중 **어느 레이어를 건드리는지** 먼저 고정한다.

## 왜 프롬프트를 잘 써야 하냐

이 주제는 `Claude Code`가 쉽게 아래 실수를 한다.

- 단일 TTS 엔진으로 모든 문제를 해결하려고 함
- 스트리밍이 아닌 batch mp3 생성으로 끝냄
- "voice cloning 지원" 문구만 보고 실제 저지연 UX 검증 없이 진행함
- 한국어 품질, TTFA, 끊김, chunk 재생 같은 실제 운영 지표를 빼먹음
- Hermes provider 레이어와 별도 TTS 서버 레이어를 섞어서 설명함

그래서 프롬프트는 반드시:
1. 목표 레이어 고정
2. 비교 기준 명시
3. 검증 기준 명시
4. 결과물 형식 고정
이 4개를 포함해야 한다.

## 제일 좋은 진행 방식

### 1단계 — 먼저 조사/설계만 시키기
처음부터 구현시키지 말고, 아래처럼 **기술 조사 + 설계안** 먼저 받는 게 좋다.

```text
작업 목표:
- tikke / tikketone / Hermes 연동 관점에서 보이스 클로닝 + 저지연 TTS 구조를 설계해줘.

중요:
- 아직 본 구현은 하지 말고, 먼저 조사/설계/후보 비교까지만 해줘.

배경:
- 내가 원하는 것은 Edge TTS처럼 채팅이 오면 거의 바로 읽기 시작하는 UX다.
- 동시에 일부 중요한 응답이나 이벤트는 보이스 클로닝된 음성으로 읽고 싶다.
- 한국어 품질이 중요하다.
- 단일 엔진으로 다 해결하는 것보다, 일반 읽기용 저지연 엔진 + 클론 전용 엔진의 2단 구조도 허용한다.

조사 대상:
- ElevenLabs
- Cartesia
- PlayHT
- MiniMax TTS
- 필요하면 로컬/오픈소스 대안도 짧게 포함

반드시 비교할 항목:
- 한국어 품질
- time-to-first-audio
- streaming / websocket 지원 여부
- instant voice cloning 가능 여부
- 구현 난이도
- 비용 예측 가능성
- Hermes custom provider 또는 command provider로 붙이기 쉬운지

산출물:
- 추천 아키텍처 2~3안
- 각 안의 장단점
- 가장 현실적인 1순위 추천안
- 이후 스파이크 구현 순서

금지사항:
- 아직 프로덕션 코드 대규모 작성 금지
- 벤더 문서 확인 없이 감으로 API 스펙 단정 금지
- '이론상 가능' 수준 설명만 하지 말고 실제 구현 가능성 기준으로 판단

결과 보고 형식:
- 추천안 A
- 추천안 B
- 추천안 C(선택)
- 1순위 추천
- 바로 다음 스파이크 작업 제안
```

## 왜 이 프롬프트가 좋은가

이 프롬프트는 `Claude Code`를 바로 구현 모드로 뛰어들게 하지 않고,
먼저 **의사결정 문서 생산 모드**로 묶는다.
이 단계가 없으면 나중에 갈아엎을 확률이 크다.

## 2단계 — 스파이크 구현을 시킬 때 좋은 프롬프트

설계가 끝나면, 다음은 **실험용 최소 구현**을 시켜야 한다.
여기서 핵심은 "완성 제품"이 아니라 **측정 가능한 prototype**이다.

```text
작업 목표:
- 저지연 TTS + 보이스 클로닝 구조의 feasibility를 확인하기 위한 스파이크를 만들어줘.

범위:
- 아직 프로덕션 통합 전체를 하지 말고, 최소 실행 가능한 prototype만 구현
- 우선순위는 '실제로 빨리 읽기 시작하는지'와 '클론 음성이 별도 분기되는지' 확인하는 것

원하는 구조:
- 일반 읽기용 엔진 1개
- 클론 전용 엔진 1개
- 텍스트 입력에 따라 어느 엔진을 쓸지 분기하는 간단한 라우터

필수 요구사항:
- 스트리밍 또는 chunk 기반 재생 구조일 것
- 전체 텍스트 완료 후 mp3 생성 후 재생하는 방식이면 안 됨
- 첫 음성 시작 시간(TTFA)을 로그로 확인할 수 있어야 함
- 짧은 일반 문장과 클론 대상 문장을 각각 테스트할 수 있어야 함
- 한국어 테스트 문장을 포함할 것

수정 대상 후보:
- [여기 실제 repo 경로 적기]
- 가능하면 spike용 별도 폴더나 script로 격리

산출물:
- 실행 방법
- 필요한 env 목록
- 테스트 입력 예시
- 측정 로그 예시
- 현재 한계

금지사항:
- 기존 메인 런타임을 크게 오염시키지 말 것
- 실제 동작 확인 없는 mock 성공 보고 금지
- 벤더 SDK를 붙였으면 최소 1회는 실행 검증할 것

검증:
- 일반 문장 1개 재생
- 클론 대상 문장 1개 재생
- 각각 첫 음성 시작 시간 기록
- 실패 시 어디서 막혔는지 명확히 남길 것

결과 보고 형식:
- 만든 파일
- 실행 명령
- 실제 실행 결과
- TTFA 관찰값
- 현재 병목
- 다음 단계 제안
```

## 왜 스파이크 프롬프트가 중요하냐

이 단계에서 확인해야 하는 건 기능 수가 아니라:
- 진짜로 빨리 말하는지
- 진짜로 클론 분기가 되는지
- Hermes에 붙일 만한 구조인지
이 3개다.

## 3단계 — Hermes 연동 구현을 시킬 때 좋은 프롬프트

스파이크가 검증되면 그 다음에야 Hermes 쪽 통합을 시키는 게 맞다.

```text
작업 목표:
- 검증된 TTS 스파이크를 Hermes에서 호출 가능한 형태로 통합해줘.

배경:
- 일반 읽기용 저지연 엔진과 클론 전용 엔진을 분리한다.
- Hermes에서는 상황에 따라 두 엔진 중 하나를 호출하게 만들고 싶다.

구현 목표:
- Hermes custom provider 또는 command provider 방식 중 더 현실적인 쪽으로 붙이기
- 설정 파일에서 엔진 선택/분기 기준을 관리할 수 있게 하기
- 최소한 일반 읽기 / 클론 읽기 두 경로를 분리

반드시 고려할 것:
- Hermes 기존 TTS 흐름과 충돌 최소화
- 설정 변경 포인트를 명확히 문서화
- fallback 전략 포함
- 실패 시 Edge 같은 기본 엔진으로 되돌릴 수 있는 안전장치

검증:
- Hermes 호출에서 일반 읽기 1회 성공
- Hermes 호출에서 클론 읽기 1회 성공 또는 실패 원인 명시
- 설정만 바꿔 롤백 가능한지 확인

결과 보고 형식:
- 수정 파일
- 설정 방법
- 실행/검증 방법
- fallback 방식
- 남은 리스크
```

## 실제로는 이렇게 나눠서 보내는 게 제일 좋다

내 추천 순서:
1. **조사/설계 프롬프트**
2. **스파이크 프롬프트**
3. **Hermes 통합 프롬프트**
4. 필요하면 **운영 최적화 프롬프트**

이 순서가 좋은 이유:
- Claude Code가 초반부터 과하게 구현하지 않음
- 실패 지점을 빨리 알 수 있음
- 벤더 선택 실수를 줄임
- 나중에 갈아엎는 비용이 줄어듦

## 한 번에 다 시키고 싶다면 이렇게 써야 함

아예 한 번에 맡기고 싶으면, 최소한 아래 정도는 넣어야 한다.

```text
작업 목표:
- tikke / tikketone / Hermes 연동을 염두에 두고, 보이스 클로닝 + 저지연 TTS 구조를 조사하고, 가장 현실적인 아키텍처를 정한 뒤, 최소 실행 가능한 spike까지 구현해줘.

우선순위:
1. 체감 지연 최소화
2. 한국어 품질
3. 보이스 클로닝 품질
4. Hermes 연동 가능성

핵심 제약:
- 단일 엔진 만능주의로 가지 말 것
- 일반 읽기용 저지연 엔진과 클론 전용 엔진 분리 가능성을 기본안으로 포함할 것
- batch mp3 생성 후 재생 방식은 저지연 목표에 부적합하므로 기본안으로 채택하지 말 것
- 실제 실행 검증 없는 성공 보고 금지

반드시 할 일:
- 후보 벤더 비교
- 추천 아키텍처 제안
- 최소 spike 구현
- 실행 방법 문서화
- 병목/한계 기록

결과 보고 형식:
- 조사 결과 요약
- 선택한 구조와 이유
- 구현한 파일
- 실행 결과
- 다음 단계
```

## 하지만 나는 이 방식보다 분할 프롬프트를 더 추천함

이유는 단순하다.
이 주제는:
- 외부 벤더 의존성 있음
- 품질 비교 필요함
- 실시간성 검증 필요함
- Hermes 연동 포인트가 따로 있음

그래서 한 번에 다 시키면 `Claude Code`가
- 조사도 대충 하고
- 구조도 성급히 정하고
- 스파이크도 애매하게 만들고
- "가능하다"고만 말할 위험이 크다.

## Claude Code에게 꼭 넣어야 하는 문장

아래 문장은 거의 필수다.

### 1. 오버엔지니어링 방지
- 프로덕션 전체 통합 전에 spike로 feasibility를 먼저 검증해줘.

### 2. 실제 검증 강제
- 실제 실행 확인 없는 성공 보고는 하지 말고, 실패하면 정확한 막힘 지점을 남겨줘.

### 3. 레이어 분리 강제
- 벤더 비교, TTS 라우팅, Hermes 통합을 한 단계로 섞지 말고 분리해서 진행해줘.

### 4. 실사용 지표 강제
- 첫 음성 시작 시간(TTFA), 한국어 품질, 끊김 여부를 반드시 보고해줘.

## 내가 추천하는 최종 프롬프트 전략

### 가장 좋은 전략
- 첫 메시지: **조사/설계만**
- 두 번째 메시지: **스파이크 구현**
- 세 번째 메시지: **Hermes 통합**

### 이유
- 이게 가장 실패 비용이 낮다.
- 그리고 `Claude Code`가 제일 덜 헛똑똑해진다.

## 내가 대신 한 줄로 요약하면

이 작업을 `Claude Code`에게 맡길 때는,
**"빠른 TTS를 만들어줘"**라고 보내면 안 되고,
**"후보 비교 → 추천 구조 결정 → 저지연 스파이크 검증 → Hermes 통합"** 순서로 강제하는 프롬프트를 보내는 게 맞다.
