# Tikke 리그 컷오프 자체수집 + 앵커

리그 조각컷(승급 컷오프)과 앵커 리그 배지를 데스크탑 없이 서버(Cloudflare Worker)에서 24/7 자체 수집하도록 만든 작업 정리. 2026-07-06.

## 핵심 결론

- 리그 컷오프 원천 = TikTok `rank_type=16`(ClassRank)의 `owner_rank.gap`. 컷오프 = 앵커 점수 + 다이아needed. 앵커 리그상태(티어/점수/순위)도 같은 owner_rank에서 나온다.
- 특별한 데이터 소스가 있는 게 아니라, 차이는 **수집 빈도/커버리지/항상 켜짐** 뿐. 그래서 tikke는 **자체 수집**으로 같은 값에 도달하는 전략이 정답이자 경쟁 우위(무료 공개 가능).

## Tikke 자체 수집 구조 (구현)

- **원천**: 라이브 앵커의 방에서 `rank_type=16` 조회 → owner_rank.gap 파싱. 앵커 1명 = 퍼센타일 1개(그 앵커가 쫓는 다음 등급 컷).
- **커버리지 핵심**: 앵커를 점수 분포 전체에서 **분산 샘플링**해야 여러 퍼센타일(1/10/20/60 등)이 다 잡힌다. 점수 top만 뽑으면 1%만 수집됨(과거 구멍).
- **발견(headless)**: TikTok `api-live/user/room`은 **무서명 공개 GET** → uniqueId로 현재 라이브 roomId 해석 가능. `liveRoom.status===2`면 라이브. Cloudflare Worker IP에서도 정상 응답(검증).
- **워커 cron 5분**: ①기존 방 풀 폴링 + 죽은 방 퇴출 ②라이브 앵커 부족 리그를 시드로 자체 발견 → 풀 등록 → 즉시 수집 ③앵커 상태 + 시드 자가치유도 같은 응답에서 write. → **데스크탑 없이 18리그 24/7**.
- **앵커 배지**: 컷오프와 같은 owner_rank에서 앵커 티어/점수 추출 → 배지가 읽는 테이블에 상시 갱신. 항상 신선.

## 데이터 함정 (반복 주의)

- 컷오프 테이블은 UNIQUE 제약이 없으면 upsert가 매번 INSERT돼 중복 폭증(과거 무료 한도 초과 사례). 제약 필수.
- 서빙은 퍼센타일별 MAX + 단조성(낮은%=높은컷) 필터로 쓰레기값 제거.
- 컷오프는 하루 중 상승만(리셋=일별), 리그 날짜는 KST(UTC+9) 기준.

## 검증 상태

- 코드/배포/헤드리스 발견 경로는 검증. **실제 라이브→수집 end-to-end**는 라이브 앵커 있는 피크 시간대에 확인 필요(오프피크엔 라이브 앵커가 없어 수집 0).
