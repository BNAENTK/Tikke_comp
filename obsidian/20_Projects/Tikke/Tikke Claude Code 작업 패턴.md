# Tikke Claude Code 작업 패턴

`Claude Code`를 `tikke` 작업에 붙일 때 반복해서 쓰는 운영 패턴.

## 기본 패턴

1. Hermes가 목표, 제약, 금지사항을 먼저 정리한다.
2. 필요한 경우 작업을 기능/버그/문서/검증 단위로 분해한다.
3. Claude Code에는 **수정 대상, 기대 결과, 검증 기준**이 보이는 프롬프트를 준다.
4. 결과가 나오면 Hermes가 검증/문서화/Obsidian 저장을 맡는다.

## 자주 쓰는 패턴

### 기능 추가 패턴
- 먼저 볼 노트: [[Tikke Context 스킬 요약]], [[Writing Plans 스킬 요약]]
- 핵심: 관련 파일 경로, UI/오버레이 연결 지점, 검증 기준을 먼저 고정
- 끝나고 확인: 실제 미리보기/링크/배포 영향

### 버그 수정 패턴
- 먼저 볼 노트: [[Systematic Debugging 스킬 요약]], [[Tikke 버그-원인 기록]]
- 핵심: 증상만 던지지 말고 재현 조건과 기대 동작을 같이 전달
- 끝나고 확인: 재현이 사라졌는지, 비슷한 경로에 회귀가 없는지 확인

### 큰 작업 분해 패턴
- 먼저 볼 노트: [[Subagent Driven Development 스킬 요약]], [[Tikke 운영 스킬 묶음]]
- 핵심: 한 번에 큰 덩어리로 던지지 말고 2~5분 단위 태스크로 쪼개기
- 끝나고 확인: 각 태스크 결과를 Hermes가 통합해서 기록

### PR/정리 패턴
- 먼저 볼 노트: [[GitHub PR Workflow 스킬 요약]], [[Requesting Code Review 스킬 요약]]
- 핵심: 구현과 리뷰를 같은 턴에 섞지 않고 마지막에 별도 점검
- 끝나고 확인: 변경 요약, 검증 결과, 남은 리스크 정리

## 프롬프트에 꼭 넣을 것

- 작업 목표
- 수정 대상 경로
- 금지사항(버전 다운 금지, 코드 원문 Obsidian 저장 금지 등)
- 검증 방법
- 결과 보고 형식

## 피해야 할 패턴

- 같은 파일을 Hermes와 Claude Code가 동시에 수정
- 맥락 없이 "고쳐"만 전달
- 오버레이 작업인데 preview/link 검증 없이 종료
- 검증 없이 결과를 바로 배포 단계로 넘김
